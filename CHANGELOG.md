## [Unreleased]

### Added
- **Auto view-tracking focus (Spark viewer).** The viewer continuously drives `spark.lodPosOverride`/`lodQuatOverride` to a virtual pose ~10% of the hit distance off whatever the screen centre is looking at (raycast the centre each ~180 ms when the camera moved), so the LoD selects the fine leaves for that region even at a far framing — the only mechanism that beats the distance-vs-LoD ceiling (a far camera never selects sub-pixel leaves no matter the budget/lodRenderScale/lodSplatScale/finer build — all exhaustively disproven). No gesture: turn the camera and the new centre sharpens itself ~0.2 s later. Gated to after preload (so it doesn't fight the preload orbit-walk), off during camera-path playback / bench / `?stock=1`; toggle at runtime with **F**. Paired with a **moderate cone foveation default** (`coneFov0 55 / coneFov 110 / coneFoveate 0.5 / behindFoveate 0.25`, non-STOCK; per-scene config and `?coneFov0=` still override) so the centre is crisp while edges stay soft-but-acceptable rather than the aggressive-global mush.
- **Toggleable on-screen settings HUD** — press **H** in any Spark viewer to show budget / active splats / fps / focus state / lodRenderScale / device tier. Off by default; replaces the ad-hoc Playwright-injected overlay (also self-labels screenshots).
- **`?bench=probe` — teleport/load benchmark with adaptive per-pose load-in measurement.** A new bench mode that teleports the camera through a list of authored poses (`viewer-config.json` → `probe_views`: `[{pos[3], quat[4], fov}]`, e.g. captured via "Set start view" at points of interest) and, at each pose, **waits until streaming actually settles** rather than holding a fixed dwell — the question being answered is "how long until all important detail is in", so a fixed timeout would have measured the wrong thing. "Settled" means VISUALLY converged, gated on true **quiescence** — `splats` *count* hitting the budget is the wrong proxy because Spark refines coarse→fine after a teleport and that swap is roughly count-neutral, so the count plateaus while the image is still sharpening (user-reported "it moves on although the image is all blurry"). The gate requires, held for `SETTLE_WIN_MS` (1500 ms): zero active fetchers **and** zero decode backlog (`pager.fetched`) **and** working-set size (`fetchPriority.length`) **and** resident-page count **and** active-splat count all unchanged — i.e. nothing in flight and the LoD selection has stopped moving. (`spark.current.mappingVersion` was tried as the gate but the v14 contact sheet disproved it — it keeps ticking on minor re-map/re-sort even when the image is visually static, so every pose hit `MAX_WAIT` while fully sharp; it is now reported informationally only, never gated on. This is exactly why the built-in screenshots exist: ground truth corrected the heuristic.) `MIN_LATENCY_MS` (400 ms) post-teleport floor blocks the pre-reaction "looks quiet" false-positive; `MAX_WAIT_MS` (16 s) cap so a pose that never resolves (e.g. phone where the LoD working set exceeds the page pool) records `settled:false` and the run continues. `fetchPriority.length` is used only as a *stability* signal, never a drain signal — it is the desired LoD working-set size and stays ~150–200 (desktop) / ~76 (phone) even when fully loaded (verified from real device traces), so a "queue→0" test would never fire. **Built-in per-pose screenshots:** the canvas is grabbed at each settled pose (synchronously inside the render loop right after `renderer.render()`, the only way to read valid pixels with `preserveDrawingBuffer:false`) and assembled into one downloaded **contact-sheet JPEG** — a labelled grid (pose # · load-in ms · splats, green=settled / red=timed-out) so the user sees exactly what "loaded" looked like at every artwork, independent of the heuristic. Each pose is pinned by a constant 2-keyframe hold player, so the mode inherits every robust-bench guarantee (camera ownership via `_player`, the `!_player`-gated `controls.update()` clobber-fix, the monotonic recorder, the camera-motion self-check, `window.__benchResult` / `bench:done`). Per-pose results (`loadInMs`, `settled`, `splats`, `pages`) are written to `_benchTrace.probe.perPose` in the downloaded JSON, mirrored to `window.__probeLoadIn` and `window.__benchResult.probe`, and `console.table`-d. Falls back to a 6-pose orbit-ring around the start view when no `probe_views` are configured, so it is never empty. `probe_views` is read but never written by the viewer (static-file constraint preserved).
- **`?bench=rotate` — frame-times-while-loading benchmark (bench #2).** Where `?bench=probe` answers "how long until loaded" (static), `rotate` answers "how janky is it while it loads as you move": it teleports to each `probe_view` and then oscillates the look ±55° around world-Y for 6 s **while the new off-axis detail streams in**, recording per-frame `dt` + longtasks + `fetchersActive`. Per-pose it reports `fps_mean`, `dt_p50/p95/p99/max`, `longtasks`, and `jankWhileFetching` (frames >50 ms *while fetchers were active* — the actual stutter-during-load metric, e.g. the M1 14 fps / 82-longtask case). The rotation path is `loop:true` with `θ=ROT_DEG·sin(2π·f)` so start==end (seamless loop, never auto-`stopPath()`s between poses). Reuses the robust recorder, the in-render-loop screenshot capture and the contact-sheet builder (`_probeSheet` generalised with an optional label formatter + `kind` for the filename) — the sheet tile turns green when `fps_mean≥50 && p99<100 ms`. Results in `_benchTrace.rotate.{fps_mean,dt_p99_max,longtasks_total,jankWhileFetching_total,perPose}`, `window.__rotateStats`, `window.__benchResult.rotate`, `console.table`. Same `probe_views` source + orbit-ring fallback; shares `_benchStop` / camera self-check (`rotate` registered as a moving mode).
- **Bench modes exposed in the viewport.** A `bench-mode` `<select>` (next to the Bench button, same styled-control pattern as the splat-budget dropdown) lets any viewer launch Orbit / Probe (load time) / Rotate (fps while loading) / Dolly-in / Cold load without a URL param — both new load benches are now one click away in-app. The dropdown is pre-synced to `?bench=` (deep links still work and stay consistent), and the Bench button runs whatever is selected; `?bench=1` still forces the static recorder.
- **Safari-on-Mac streaming hint (Spark viewer) — the shipped mitigation for the ANGLE-Metal jitter (#78).** A small, dismissible in-viewer notice ("For the smoothest playback on Mac, open this scene in **Safari** — other browsers can stutter while detail streams in"). Shown **only** when the live WebGL renderer string contains both `ANGLE` and `Metal` — i.e. a Chromium/Firefox browser on a Mac, the *exact* discriminator that proved the unfixable synchronous ANGLE→Metal staged-upload jitter (Safari → `Apple GPU`, no ANGLE; Windows/Linux ANGLE → no `Metal`; iOS → all WebKit, no ANGLE — every non-affected platform is correctly excluded). A conservative UA fallback (Mac-desktop + non-Safari engine) covers only the case where the renderer string is masked (Brave strict shields / Firefox RFP). Dismissal is remembered via `localStorage` (no nagging every load); `?stock=1` opts out; `?safariHint=1|0` forces it on/off for verification on any machine. Reuses the renderer string the Apple-Silicon profile already captured (no extra GL context). This is the conclusion of the exhaustive #78 root-cause: the jitter is intrinsic to ANGLE-Metal on Apple Silicon and unfixable from the app or via any flag (3 fork upload rewrites + WebGPU + ANGLE's own `alwaysPreferStagedTextureUploads` off-switch all measured ~83–86 longtasks; Safari native = 0) — so the honest fix is to point Mac visitors at the browser that *is* smooth.
- **Shareable link preview cards (Open Graph + Twitter) for the Spark viewer.** Pasting a deployed viewer URL into Telegram / WhatsApp / iMessage / Discord / Slack / Twitter / LinkedIn now unfurls a rich card (title · description · large image) instead of a bare link. `template.py`'s `html_for()` gained optional `share_url`, `share_image`, and `description` kwargs and emits a `{share_meta}` block of `og:*` + `twitter:card=summary_large_image` + `<meta name=description>` tags in `<head>` (all dynamic text HTML-escaped; the joined block is substituted as one `.format()` field so no brace-doubling and no injection risk). Fully backward-compatible — every kwarg is optional, so `SparkAssembler` and older deploy scripts keep working and still get a title/description card; `share_image` defaults to a relative `preview.jpg` (a scraper resolves it against the page URL) while deploy scripts pass an **absolute** Bunny URL for maximum cross-platform reliability, and `og:url` is emitted only when `share_url` is given. The card image is a 1200×630 screenshot of the scene at its start view, generated and uploaded as `preview.jpg` next to `index.html` by a new reusable `.codex-run/make_share_preview.py` (Playwright loads the deployed viewer, hides the HUD/hint/loading chrome, captures, uploads + purges) so every scene — including the #57 redeploys — gets a card automatically with no backend (fits the static-CDN constraint).

### Changed
- **Spark desktop default budget 1.5 M → 2 M** (phone 500 K / tablet 1 M unchanged). With auto-focus concentrating the budget on the looked-at region, 2 M is enough to keep the centre sharp on an M1-Pro-class machine (validated: 2 M + focus ≈ 6 M + focus on the IBUG wall, 113 fps).
- **Spark 2 root-chunk pin replaced with a page-eviction guard — fixes slow detail streaming.** The viewer kept coarse coverage resident (so a not-yet-streamed region never renders as a blank gap) by re-prepending 16 root chunks to the *front* of the pager's `fetchPriority` on **every** `driveFetchers()` call. With only 3–4 fetch slots that made fine, camera-relevant chunks queue behind 16 coarse ones every frame, so detail "loaded in" slowly and LoD transitions crawled visibly ("reshading"). Verified against `spark/src/SplatPager.ts`: the new approach (a) queues a still-missing root chunk once, appended at the *end* so Spark's own camera-priority ordering keeps full priority + all fetch slots for in-view detail, and (b) filters root-chunk pages out of `freeablePages` after the real `driveFetchers` so `allocateFreeable()` can never evict them — the actual anti-disappear guarantee, at zero fetch-priority cost. Measured: cold-start preload ~30% faster (Speicher 2193 ms vs ~3100 ms; Stettiner 3201 ms vs ~3300–3900 ms). Regression-checked: no splat-disappear under aggressive orbit + deep zoom on Speicher (the disappear-prone scene) and Stettiner (30 M, the largest); 0 console errors. Also corrected the false "Spark numLodFetchers hard cap is 4" code comment (that is the decode-worker pool; fetch concurrency is not capped).
- **Spark 2 `numLodFetchers` raised and made tier-aware: desktop 8, tablet 6, phone 3** (was 4 / 4 / 2). With the per-frame coarse-chunk re-prepend gone (above), the fetch slots are no longer wasted re-confirming resident roots, so more parallel HTTP-Range fetchers directly hide CDN round-trip latency and stream fine detail in faster. Decode is still bounded by Spark's fixed 4-worker pool (hence diminishing returns past ~8). Corrected a false code comment that claimed Spark's `numLodFetchers` is hard-capped at 4 — verified in `spark/src/SplatPager.ts` that the default is 3 and uncapped; the 4 is the decode pool. Regression-checked on Speicher + Stettiner (30 M): no splat-disappear under aggressive orbit/zoom, 0 console errors.
- **Spark 2 desktop `lodRenderScale` lowered to `0.75`** (from Spark's default `1.0`; phone/tablet keep `1.3`), plus a new `?lodRenderScale=N` URL override (highest priority, same spirit as `?budget=`). Rationale: at the default `1.0` the LoD never subdivides into sub-pixel splats, so during close/mid navigation the visible count plateaus below the chosen budget even when finer detail exists; `0.75` lets it climb toward the budget there. Measured caveat (verified on Speicher, RTX desktop): at wide *aerial* framings this changes nothing — the visible count is bounded by how much geometry is on screen (~4.2 M for Speicher's spawn view), not by the detail floor, so no LoD knob raises it without moving the camera closer. It is not a bug that a 6 M budget shows ~4 M on a wide shot. Sub-pixel splats cost some fragment work; desktop-only, opt-out via `?lodRenderScale=1` or `?stock=1`.
- **Spark 2 viewer now uses a static per-device splat budget — no adaptive FPS controller.** The runtime `lodSplatScale` loop (refresh-rate detection → vote-based up/down scaling) was removed: research confirmed no shipping splat viewer uses an adaptive FPS loop, and the runtime `· NNN%` / `frozen` readout was confusing without buying real quality. `spark.lodSplatCount` is now picked once from the device tier and held: **phone 500 K, tablet 1 M, desktop 1.5 M** (measured-comfort values — iPhone 13 mini ≈ 40 fps, M1 Pro ≈ 30 fps, discrete GPUs 60 fps; the dropdown still overrides at runtime). The HUD is simplified to `Splats: X.XM · NN fps`. `maxPagedSplats` (total resident pool) is capped at **6 M** on desktop/tablet (was 32 M on desktop — needless memory for a 1.5 M visible budget) and stays at 4 M on phone for iOS canvas-memory headroom. The runtime budget dropdown is trimmed to match the pool — `500K · 1M · 1.5M · 2M · 3M · 4M · 6M` (the old `8M–23M` and `No limit` options offered counts the 6 M pool can never satisfy, which is exactly the kind of confusing knob this change set out to remove; `?budget=N` still works as a power-user escape hatch). Bench mode, orbit prefetch, the preload bar and the mobile knob set are unchanged. The `?adaptive=` and `?targetFps=` URL params are gone (`?budget=N`, `?stock=1`, `?preload=0`, `?bench=…` remain). All five live Spark viewers (IBUG, Speicher, Stettiner, Fabrik, Polygraf) redeployed.
- **PlayCanvas CDN bumped from `2.17.0` to `2.18.1`** in all three viewer surfaces — the embedded `_VIEWER_TEMPLATE` in `lod_assembly.py` (assembled-output `index.html`), the static dashboard preview `web/static/viewer.html`, and the scene editor `web/templates/scene_editor.html`. v2.18.0 was a major GSplat refactor (50+ commits): compute-based tiled WebGPU GSplat renderer, GPU radix-sort optimisations, always-on frustum culling moved into compute, vectorised Gaussian eval across 2×2 pixel quads, per-node SH color updates with angle threshold, and a new `GSplatParams.renderer` enum. The 2.17.1 / 2.17.2 update-loop regression that originally pinned us to 2.17.0-exact is gone — verified by running both versions of the same Gutsmutstrasse splat side-by-side (`tests/manual/pc-compare.html`); 2.18.1 renders 2.99 M splats with zero console errors. Also fixed a pre-existing inconsistency where `viewer.html` and `scene_editor.html` had pinned the main `playcanvas` import but loaded `camera-controls.mjs` from an unpinned `playcanvas/scripts/...` URL — both are now pinned to `@2.18.1` in lockstep.
- **`@playcanvas/splat-transform` bumped to `^2.0.4`** (was `^1.10.2`). v2.0 refactored progress reporting (PR #204) — the v1.x `[1/8] Generating morton order` per-chunk step labels are gone, replaced by a `▸`-prefixed structured tree. The `lod_assembly` progress parser now matches the v2.x chunk lines (`▸ [N/M] X_Y` inside the Writing section) via `_CHUNK_RE`, with `_SECTION_RE` capturing top-level section labels (Reading / Writing / Filter NaN) for the user-visible step message. Both the streaming and non-streaming command builders now pass `--no-tty` so the new logger emits stable, line-by-line output suitable for subprocess piping. Captured-stderr unit tests in `test_lod_assembly.py::TestProgressRegex` lock the parser against the v2.x format.
- **`pyproject.toml` `[tool.ruff] target-version`** updated from `py311` to `py312` to match the Python floor bumped in v0.7.0.
- **`--rad-chunked` is now the Spark 2 pipeline default.** `build_lod.build()` defaults to `chunked=True`: build-lod emits a small manifest `<stem>-lod.rad` plus N sibling `<stem>-lod-<i>.radc` chunk files instead of one monolithic `.rad`. The set is cached as a *directory* under `~/.cache/splatpipe/rad/<key>/` — staged via a `.tmp` sibling then atomically swapped, so a crashed build never leaves a half-written cache that the new `_find_manifest()` completeness guard would wrongly serve; `build()` returns the manifest path. `SparkAssembler` copies the manifest as `scene.rad` and every `.radc` with its **original basename** (the manifest references chunks by basename, resolved by the viewer relative to the manifest's own URL — no viewer or `PRIMARY_ASSET` change needed). `deploy.py` already handled the set (`rglob` upload + a pre-existing `.radc` → `max-age=31536000, immutable` rule + purge). Why: hundreds of independently CDN-cacheable objects (IBUG = 477) kill the cold-edge Range-miss, parallelise over the fetchers + HTTP/2, and let repeat visitors hit warm per-file edge cache — measured ~1.5× faster first paint. `splatpipe build-lod` gains `--chunked/--no-chunked` (default chunked) and now reports total size + chunk count instead of the tiny manifest's size. The legacy single-file path is fully preserved via `chunked=False` / `--no-chunked`. New `tests/test_spark_chunked.py` (9 tests) locks the cache-completeness guard and the chunked/non-chunked cache layout + key isolation.
- **`--cluster-sh` is now the Spark build default (large-scene pipeline productised).** This whole session converged on one way to ship large Gaussian scenes; it is now the splatpipe default, not a one-off `.codex-run` script. `build_lod.build()` gained `cluster_sh: bool = True` — it passes build-lod `--cluster-sh` (SH coefficients vector-quantised into a ≤64K codebook): empirically **~60% smaller `.rad`** (IBUG 1653 MB → 661 MB) at no perceptible quality loss and far less GPU/JS memory, which is what makes 3–7 GB source PLYs deployable at all. `SparkAssembler` now builds cluster-sh by default; `splatpipe build-lod` gains `--cluster-sh/--no-cluster-sh` (default on). The flag set (`q`/`n` + `c` + `s` + extra) is part of the build cache key so flipping it rebuilds correctly rather than serving a stale asset. **Hard dependency, also already default:** cluster-sh `.rad`s only run on the patched self-hosted Spark fork the viewer template pins via `SPARK_FORK_URL` (rcf2 — the `ChunkDecoder` RefCell-reentrancy fix + the chunk-0 SH-codebook ordering gate; upstream `@sparkjsdev/spark@2.0.0` panics/OOMs on every cluster-sh chunk). `--no-cluster-sh` produces a larger, stock-Spark-compatible build for anyone not using the fork. Reusable `.codex-run/deploy_scene_cs.py` builds+deploys any source PLY through this default pipeline (chunked cluster-sh on the final template: share-card, per-scene clip_xy, #81, rcf2) to a fresh Bunny path.
- **Spark `clipXY` is now per-scene config, default lowered 3.0 → 1.4 (Phase 1 #6).** `clipXY` (per-splat XY frustum-cull slack) was a hard-coded global `3.0` in `template.py` — a margin only a few scenes actually need (it was bumped globally solely because Speicher's `.rad` has giant outlier splats, ln-scale ≈9, that Spark's default 1.4 culls when their centres go off-screen → "scene goes blank"). The wide margin is pure fragment cost for every other scene. Now `spark_render.clip_xy` (added to `Project.DEFAULT_SCENE_CONFIG.spark_render` and the viewer `_DEFAULTS`, default **1.4** = Spark's own default) is read per scene from `viewer-config.json`, with a new `?clipXY=N` URL override joining the existing A/B detail-lever knobs. **Migration:** a scene that genuinely needs the wide margin (Speicher, and any other with training-outlier splats) must carry `spark_render.clip_xy: 3.0` in its own `viewer-config.json` / project `scene_config`; this is injected at that scene's redeploy. Scenes redeployed without it use 1.4 — verified on IBUG (no blanking under aggressive orbit/zoom) before shipping.

### Added
- **"Set start view" — author a Spark scene's opening camera from inside the deployed viewer.** New header button captures the live camera (position, rotation, FOV, orbit target), shows a confirm dialog, then emits a compact `SPV1:<project>:<base64url>` token (copied to clipboard + shown selectable). Because the viewers are static files on Bunny with no backend (and the storage key must never be in client JS), persistence is a relay: the user sends the token to the assistant, who runs the new **`splatpipe set-start-view "<token>"`** CLI. That decodes/validates it and writes a `start_view` field into the project's `viewer-config.json` on Bunny (+ CDN purge; and into the local project's `scene_config` when `--project-path` is given, so re-exports keep it). The viewer treats `start_view` as the **highest-priority** initial camera — above camera-paths, annotations and the auto sqrt-distance fallback — and it seeds the Reset/Home + orbit-bench origin too. `start_view` added to `Project.DEFAULT_SCENE_CONFIG`. Verified end-to-end on Bunny (button → token → CLI → reload spawns at the saved pose). Design: `docs/plans/2026-05-15_set-start-view.md`.
- **Fixed: `viewer-config.json` is now fetched with `cache: 'no-store'`.** Bunny's pull zone serves it with a 30-day `max-age` (its default override; "Honor Origin Cache-Control" is off so `.rad` stays long-cached), so a browser that had already opened a viewer kept a stale scene config for weeks — which would have silently broken "saved for all" for `start_view` and any scene-config edit. The file is a few KB; always revalidating it is free and makes config changes appear on the next load.
- **Manual mobile test harness for PlayCanvas bumps** under `tests/manual/`: `pc-2-18-1.html` is the deployed Gutsmutstrasse viewer with the playcanvas pin swapped, and `pc-compare.html` is a split-pane shell (vertical on phones, side-by-side on landscape ≥800px) iframing the production viewer next to the test page. Designed so the next PlayCanvas bump is a 30-second version-string edit + Bunny re-upload, not a fresh detective story. Lesson learned the hard way: PlayCanvas resolves chunk URLs from `lod-meta.json` relative to the *page URL*, not `<base href>`, so the test page has to live in the same folder as the chunks.
- **LichtFeld PPISP train option in the project UI.** The Train Splats settings now show Photometric Compensation: it is enabled for LichtFeld projects and maps to the existing `--ppisp` trainer flag, while Postshot projects show the control as unavailable because Postshot CLI v1.0.359 does not expose a photometric-compensation flag.

### Fixed
- **`--cluster-sh` Spark builds no longer crash the viewer — patched, self-hosted Spark fork.** Upstream `@sparkjsdev/spark@2.0.0` panics `RefCell already borrowed` in `ChunkDecoder::push` (`rust/spark-worker-rs/src/decoder.rs`) on every chunk of a `--cluster-sh` build: the function held a `borrow_mut` on the `BUFFER` thread-local `RefCell<Vec<u8>>` across the `self.receiver.push()` callback, and the cluster-sh receiver path (SH-codebook + `sh_label` expansion) re-enters that same thread-local — the nested borrow panics, aborting the WASM module, which the browser surfaces as a generic "Out of Memory" page even with tens of GB of RAM/VRAM free. Plain (non-cluster) chunked builds never re-enter, which is why only `--cluster-sh` crashed despite being ~60% smaller (IBUG 661 MB vs 1653 MB). Forked at `H:/001_ProjectCache/1000_Coding/spark` (branch `fix/chunkdecoder-refcell-reentrancy`): `mem::take` the scratch `Vec` out of the `RefCell` so it is not borrowed while the receiver runs, then restore it for reuse — preserves the per-thread buffer-reuse optimisation, zero decode-behaviour change. The patched production ESM bundle (`vite --mode production` → `spark.module.min.js`, WASM base64-inlined, `three` left as a bare import resolved by the page importmap) is self-hosted on Bunny at a version-pinned, never-reused path (`/_sparkfork-rcf1/spark.module.min.js`; bump `-rcfN` per fork change for edge-cache safety). `template.py` gains a `SPARK_FORK_URL` constant and repoints the `@sparkjsdev/spark` importmap entry off jsdelivr to the fork — Spark is now same-origin with the viewer HTML + `.rad`/`.radc` (connection already warm); only `three` still preconnects to jsdelivr, and the preconnect comment is corrected accordingly. Root cause was captured live from the failing viewer's console via Playwright after several disproven memory/VRAM/V8-2 GiB-cap theories — the actual fault is a Rust re-entrancy panic, not an allocation failure.
- **`--cluster-sh` second abort fixed — chunk-0 SH-codebook ordering gate.** Even with the RefCell fix above, `IBUG_cs_v2` still aborted ~7×/load with `panicked at spark-worker-rs/src/packed_splats.rs:695: index out of bounds: the len is 0 but the index is N` — caught live via Playwright during the Phase 0 measurement pass (exactly why measurement precedes optimisation). Root cause: `--cluster-sh` carries the SH codebook only in chunk 0, but nothing sequences chunk 0 ahead of chunks 1..N; a non-zero chunk whose decode worker resolves before chunk 0's reads `this.sh1Codes/sh2Codes/sh3Codes` while still `undefined`, so Rust `set_sh_labels` indexes an empty codebook `Vec` and the WASM worker panics. Non-deterministic (depends on fetch/decode scheduling) — which is why it "felt better but wasn't fixed." Fix (`SplatPager.ts`, same fork branch): a per-`PagedSplats` `codebookReady` promise; `fetchDecodeChunk` detects a clustered build via the new `RadMeta.shCodeCount` and, for any `chunk !== 0`, awaits chunk 0's codebook before dispatching the worker decode. Plain (non-cluster) chunked builds have `shCodeCount` absent → gate skipped → zero behaviour change or regression. TS-only (no WASM rebuild; the RefCell-patched WASM is preserved). Self-hosted bundle bumped `_sparkfork-rcf1` → `_sparkfork-rcf2`.
- **Desktop Spark refetch storm fixed — page pool now covers the LoD working set.** Phase-0 instrumentation (Playwright on IBUG_cs_v3, 2026-05-16) found the real "detail loads slow / cluster-sh not faster" cause: `SplatPager.driveFetchers` evicts and **re-decodes** every chunk the per-frame-rebuilt `fetchPriority` lists beyond `maxPages`. At the 2 M desktop budget the LoD traversal selects ~182 chunks but `maxPagedSplats=6 M` gave only 92 pages, so ~90 chunks were perpetually evicted and re-decoded in the 4-worker WASM pool — **~53–79 fetch+decode/s with a *static* camera, 14× refetch over a 477-chunk scene** (`fetched`-backlog 0, fetchers avg 2.1/8 → neither decode-pool- nor fetch-bound; pure evict/re-decode thrash). Controlled live A/B (lodSplatCount sweep): 0.5 M → 79 chunks ≤ 92 → 0.3/s (storm gone); 1 M → 115; 1.5 M → 155; 2 M → 182 ≫ 92 → 79/s; `lodRenderScale` had **no** effect on working-set size (refuting a prior hypothesis). Fix (`template.py`): desktop `maxPagedSplats` 6 M → 16 M (92 → 244 pages, ~1.34× the static working set + camera-movement turnover headroom); phone stays 4 M (iPhone Safari 256 MB ceiling) and tablet 6 M (follow-up tracked) so neither memory profile regresses. Largest SH ArrayBuffer at 244 pages ≈ 244 MB, far under V8's ~2 GiB cap. No quality change (budget unchanged at 2 M). The prior "6 M is plenty" cap (2026-05-15) predated this measurement; this supersedes it on evidence.

## [0.6.3] - 2026-04-17

### Fixed
- **Max plugin Stand-Up Parent setup — wrong inner rotation in 3ds Max only.** In Max, assigning `inner.parent = outer` preserves the world transform, so setting `inner.rotation = +90°X` *before* parenting (the previous order) left `inner.world = R_+90X` instead of the documented composed `R_-90X`. Two visible symptoms: (a) the splat appeared upside-down in the Max viewport, (b) the v0.6.2 export math gave wrong PC coords (DCC `(0, 0, 5)` came back as PC `(0, -5, 0)` instead of `(0, 5, 0)`, mirrored on Y). ## [0.6.2] - 2026-04-17

### Fixed
- **DCC bridge math — Stand-Up Parent compose was missing the final 180°-X flip.** Cameras animated in 3ds Max or Blender came back mirrored on Y: a camera placed at DCC `(0, 0, 5)` (5 units above the splat) was returning as PC `(0, -5, 0)` (5 units *below*) instead of `(0, 5, 0)`.
